Here is a step-by-step explanation of how a modern web browser validates a TLS (Transport Layer Security) certificate when you visit a secure website (HTTPS).

### 1. The Handshake and Certificate Delivery
When you type a URL and hit enter, the browser initiates a "TLS Handshake" with the web server. As part of this process, the server sends its **SSL/TLS Certificate** to the browser. This certificate acts like a digital passport, containing:
*   The domain name (e.g., `google.com`).
*   The server’s public key.
*   The identity of the issuer (Certificate Authority).
*   The validity dates.

### 2. Chain of Trust Verification
The browser does not just look at the certificate in isolation; it checks the **Chain of Trust**.
*   **Intermediate Certificates:** Most servers send an "Intermediate Certificate" rather than the root certificate. The browser checks if this intermediate was signed by a trusted root.
*   **Root Store:** The browser (or the operating system) has a built-in list of "Trusted Root Certificate Authorities" (CAs). It verifies that the chain of signatures leads back to one of these trusted roots.

### 3. Validity Period Check
The browser checks the **Not Before** and **Not After** dates. If the current time is outside this window (the certificate has expired or isn't valid yet), the browser will block the connection.

### 4. Revocation Status Check
The browser must ensure the certificate hasn't been revoked (e.g., if the server was hacked or the private key was stolen). It does this via:
*   **CRL (Certificate Revocation List):** Checking a list of revoked certificates published by the CA.
*   **OCSP (Online Certificate Status Protocol):** Asking the CA directly if the certificate is still good.
*   **OCSP Stapling:** The server provides a time-stamped, signed "proof" from the CA that the certificate is valid, which is faster and more private than the browser asking the CA directly.

### 5. Domain Name Matching (SAN Check)
The browser checks if the domain you are visiting matches the name on the certificate.
*   Modern certificates use **Subject Alternative Names (SANs)** to list all valid domains.
*   If you visit `example.com` but the certificate is only for `www.example.com` (and `example.com` isn't listed in the SANs), the browser will show a warning.

### 6. Key Usage and Algorithm Check
The browser ensures the certificate is being used for its intended purpose (e.g., web server authentication) and that the cryptographic algorithms used (like RSA or ECDSA) are still considered secure and not deprecated.

### 7. The Result
*   **Validation Successful:** The browser generates a "pre-master secret," encrypts it with the server's public key, and establishes an encrypted tunnel. You see the padlock icon in your address bar.
*   **Validation Failed:** The browser halts the connection and displays a scary warning page (like `NET::ERR_CERT_AUTHORITY_INVALID` or `CERT_DATE_INVALID`), advising you not to proceed.
