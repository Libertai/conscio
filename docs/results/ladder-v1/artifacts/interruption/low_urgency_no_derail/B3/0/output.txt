TLS (Transport Layer Security) certificate validation is the process by which a browser verifies the identity of a website to ensure you are connecting to the legitimate server and not an imposter. This process establishes a secure, encrypted channel for data exchange.

Here is the step-by-step breakdown of how this validation works:

### 1. The TLS Handshake (Initiation)
Before any data is exchanged, the browser and the server perform a "handshake" to agree on encryption methods and verify identity.
*   **Client Hello:** The browser sends a message to the server containing the TLS version it supports, a list of supported encryption algorithms (ciphers), and a random number.
*   **Server Hello:** The server responds with its chosen TLS version, cipher, and its own random number. Crucially, it also sends its **SSL/TLS Certificate**.

### 2. Certificate Inspection
Once the browser receives the certificate, it begins a rigorous inspection process. The certificate is a digital document that contains:
*   The domain name (e.g., `www.example.com`).
*   The server's public key.
*   The expiration date.
*   The digital signature of the Certificate Authority (CA) that issued it.

### 3. Chain of Trust Verification
The browser does not just trust the server's certificate; it verifies the "Chain of Trust."
*   **Root CA Check:** The browser checks if the certificate was issued by a Certificate Authority (CA) that is trusted. Browsers come pre-installed with a list of trusted Root CAs.
*   **Intermediate Certificates:** Most certificates are not signed directly by a Root CA but by an "Intermediate CA." The browser checks if the Intermediate CA is signed by a trusted Root CA.
*   **Validation:** If the chain is unbroken and leads back to a trusted Root CA, the certificate is considered authentic. If the chain is broken or the CA is unknown, the browser will display a security warning (e.g., "Your connection is not private").

### 4. Domain Name Validation
The browser checks if the certificate is valid for the specific domain you are visiting.
*   **Common Name (CN) / Subject Alternative Name (SAN):** The browser compares the domain in the URL (e.g., `www.example.com`) with the names listed in the certificate.
*   **Mismatch:** If you visit `www.example.com` but the certificate is issued for `www.other-site.com`, the validation fails.

### 5. Revocation Status Check
The browser checks if the certificate has been revoked before its expiration date (e.g., if the private key was compromised).
*   **CRL (Certificate Revocation List):** The browser may download a list of revoked certificates from the CA.
*   **OCSP (Online Certificate Status Protocol):** The browser may contact the CA directly to ask if the certificate is still valid.
*   **OCSP Stapling:** To speed this up, the server often "staples" a time-stamped, signed response from the CA to the handshake, proving the certificate is valid without the browser needing to contact the CA separately.

### 6. Public Key Exchange and Key Generation
If all the above checks pass:
*   The browser generates a **pre-master secret** (a random number) and encrypts it using the server's **public key** (from the certificate).
*   The browser sends this encrypted pre-master secret to the server.
*   Only the server can decrypt it using its **private key**.
*   Both the browser and the server now use this shared secret to generate a **session key**.

### 7. Secure Communication Established
*   The session key is used to encrypt all subsequent data exchanged between the browser and the server.
*   The browser displays a padlock icon in the address bar, indicating the connection is secure.

### Summary of Key Security Principles
*   **Authentication:** Ensures you are talking to the real server.
*   **Integrity:** Ensures data hasn't been tampered with during transit.
*   **Confidentiality:** Ensures only the browser and server can read the data.

If any step fails (expired certificate, wrong domain, untrusted CA, revoked certificate), the browser will block the connection or show a prominent warning to protect the user.

Backup naming conventions are critical for data recovery, automation, and organization. Here are some widely accepted best practices and common patterns:

### 1. Key Components of a Good Backup Name
A robust backup filename should generally include:
*   **Source/Identifier:** What is being backed up? (e.g., `db`, `files`, `server01`)
*   **Timestamp:** When was it created? (Ideally in ISO 8601 format for easy sorting: `YYYYMMDD_HHMMSS`)
*   **Type/Incremental Flag:** Is it a full backup, incremental, or differential? (e.g., `full`, `incr`, `diff`)
*   **Version/Hash (Optional):** For integrity verification or version control.

### 2. Common Naming Patterns

#### A. Simple & Human-Readable
Good for manual backups or small-scale systems.
```text
backup_db_20231027_140000.sql
myfiles_full_2023-10-27.tar.gz
```

#### B. Structured & Machine-Readable (ISO 8601)
Best for automation scripts and chronological sorting.
```text
server01_full_20231027T140000Z.tar.gz
db_incremental_20231028_020000.sql.gz
```

#### C. With Retention/Versioning
Useful when keeping multiple versions of the same backup.
```text
backup_20231027_v1.tar.gz
backup_20231027_v2.tar.gz
```

### 3. Best Practices

*   **Use ISO 8601 Dates:** `YYYYMMDD` or `YYYY-MM-DD` ensures that files sort correctly chronologically when listed alphabetically.
*   **Avoid Spaces:** Use underscores (`_`) or hyphens (`-`) instead of spaces to prevent issues in scripts and command-line tools.
*   **Include Compression Extension:** Clearly indicate the compression format (e.g., `.gz`, `.bz2`, `.xz`, `.zip`).
*   **Keep it Consistent:** Stick to one naming scheme across all systems to avoid confusion.
*   **Add Checksums (Optional):** Some systems append a hash (e.g., `.sha256`) for integrity verification.

### 4. Example Workflow
If you are backing up a database daily:
*   **Full Backup (Sunday):** `db_full_20231029_030000.sql.gz`
*   **Incremental (Mon-Sat):** `db_incr_20231030_030000.sql.gz`
