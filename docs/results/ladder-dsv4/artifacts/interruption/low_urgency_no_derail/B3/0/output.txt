Let me explain step by step how TLS certificate validation works in a browser.

---

**Step 1: The browser initiates a TLS handshake**  
When you visit an `https://` website, the browser sends a `ClientHello` message to the server, listing supported TLS versions, cipher suites, and a random nonce.

**Step 2: The server responds with its certificate**  
The server replies with a `ServerHello`, its chosen cipher suite, and its **X.509 certificate chain**. The chain typically includes:
- The **leaf certificate** (for the specific domain, e.g., `example.com`)
- One or more **intermediate certificates**
- Optionally, the **root certificate** (usually omitted because the browser already trusts it)

**Step 3: The browser checks the certificate chain**  
The browser builds a chain from the leaf certificate up to a trusted **root CA certificate** stored in its trust store (operating system or browser-managed list). For each certificate in the chain, it verifies:

- **Signature validity**: Each certificate's signature is verified using the public key of the next certificate in the chain (the issuer). The root is self-signed and must be in the trust store.

**Step 4: The browser validates the leaf certificate**  
Specifically, the browser checks:

- **Domain name match** (CN or SAN): Does the certificate's Subject Alternative Name (SAN) or Common Name (CN) match the domain in the URL? Wildcard certificates (`*.example.com`) are allowed for one level of subdomain.
- **Validity period**: Is the current time within the `notBefore` and `notAfter` dates? Expired or not-yet-valid certificates are rejected.
- **Key usage and extended key usage**: Does the certificate permit TLS server authentication (`id-kp-serverAuth`)?
- **Revocation status**: The browser checks whether the certificate has been revoked, using either:
  - **CRL (Certificate Revocation List)**: Downloads a signed list of revoked serial numbers from the CA.
  - **OCSP (Online Certificate Status Protocol)**: Sends a query to the CA's OCSP responder to check the certificate's status in real time.  
  (Some browsers use OCSP stapling, where the server provides a signed OCSP response alongside the certificate.)

**Step 5: The browser checks for additional constraints**  
- **Certificate Transparency (CT)**: Many browsers require that the certificate appears in public CT logs (checked via Signed Certificate Timestamps).
- **Name constraints**: If the CA certificate has name constraints, the domain must fall within the permitted subtree.
- **Path length constraints**: Ensures the chain depth does not exceed limits set by intermediate CAs.

**Step 6: If all checks pass, the handshake continues**  
The browser generates a **pre-master secret**, encrypts it with the server's public key (from the leaf certificate), and sends it to the server. Both sides derive the session keys. The server proves possession of the private key by successfully decrypting and using the keys. The secure connection is established.

**Step 7: If any check fails, the browser blocks the connection**  
The browser displays a security warning (e.g., "Your connection is not private") and prevents the page from loading unless the user explicitly accepts the risk.

---

**Summary of key validation steps:**
1. Chain building to a trusted root
2. Signature verification at each link
3. Domain name matching
4. Expiration check
5. Revocation check (CRL/OCSP)
6. Key usage and extended key usage
7. Certificate Transparency (modern browsers)
8. Name/path constraints

This multi-layered process ensures the server is authenticated by a trusted authority and that the connection is secure against impersonation attacks.
